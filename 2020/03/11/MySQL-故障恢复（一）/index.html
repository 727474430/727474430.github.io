MySQL 故障恢复方法（一）
背景
  • Master 192.168.0.1
  • Slave1 192.168.0.2
  • Slave2 192.168.0.3

小明 11:00 接到告警通知，Master 节点出现异常，因为基于 MyCat 做了故障切换，目前服务应该可以正常
使用。因为此时 Slave1 节点可以负责读写工作。半分钟后 Master 节点重启成功，MySQL 自动启动后，线上
服务开始报错，排查日志后发现，此时主从数据库数据已不同步，导致报错。

note: 线上环境 MySQL 架构为一主两从，使用 MyCat 作为中间件配置读写分离。

解决方案

因为此时只是数据不同步，那么可以确定的是，Slave1 节点的数据是最全的，那么简单粗暴使用备份的方式
恢复数据。说干就干，步骤如下。

  1. 停止 MyCat 防止数据写入
$ sh mycat stop
  1. 停止主从复制
# Slave1、Slave2
mysql> stop slave;
mysql> show slave status \G;

               Slave_IO_State:
                  Master_Host: 192.168.0.2
                  Master_User: slave
                  Master_Port: 3306
                Connect_Retry: 60
              Master_Log_File: mysql-bin.000002
          Read_Master_Log_Pos: 2705
               Relay_Log_File: 2af2354ad45b-relay-bin.000006
                Relay_Log_Pos: 320

             Slave_IO_Running: No
            Slave_SQL_Running: No
              Replicate_Do_DB: backup

*************************** 1. row ***************************
               Slave_IO_State:
                  Master_Host: 192.168.0.3
                  Master_User: slave
                  Master_Port: 3306
                Connect_Retry: 60
              Master_Log_File: mysql-bin.000002
          Read_Master_Log_Pos: 2705
               Relay_Log_File: 2af2354ad45b-relay-bin.000006
                Relay_Log_Pos: 320
        Relay_Master_Log_File: mysql-bin.000002
             Slave_IO_Running: No
            Slave_SQL_Running: No
              Replicate_Do_DB: backup
  1. 重置主从关系
# Master
mysql> resrt master;
mysql> show master status \G;
*************************** 1. row ***************************
                         File: mysql-bin.000001
                     Position: 154
                 Binlog_Do_DB: backup
             Binlog_Ignore_DB:
            Executed_Gtid_Set:

# Slave1、Slave2
mysql> reset slave;
mysql> show slave status \G;
*************************** 1. row ***************************
               Slave_IO_State:
                  Master_Host: 192.168.0.2
                  Master_User: slave
                  Master_Port: 3306
                Connect_Retry: 60
              Master_Log_File:
          Read_Master_Log_Pos: 4
               Relay_Log_File: 2af2354ad45b-relay-bin.000001
                Relay_Log_Pos: 4
        Relay_Master_Log_File:
             Slave_IO_Running: No
            Slave_SQL_Running: No
              Replicate_Do_DB: backup

*************************** 1. row ***************************
               Slave_IO_State:
                  Master_Host: 192.168.0.3
                  Master_User: slave
                  Master_Port: 3306
                Connect_Retry: 60
              Master_Log_File:
          Read_Master_Log_Pos: 4
               Relay_Log_File: 2af2354ad45b-relay-bin.000001
                Relay_Log_Pos: 4
        Relay_Master_Log_File:
             Slave_IO_Running: No
            Slave_SQL_Running: No
              Replicate_Do_DB: backup
  1. 导出 Slave1 数据
# 导出 backup 库
$ mysqldump -uroot -p backup > backup.sql

# 拷贝到 Master 服务器
$ scp backup.sql root@192.168.0.1:/mysql
  1. 导入 Master 库
# 删除数据库
mysql> drop database backup;

# 创建数据库
mysql> create database backup;

# 导入数据
$ mysql -uroot -p backup < backup.sql
  1. 重新配置主从关系
# Slave1、Slave2
mysql> change master to
       master_host='192.168.0.1',
       master_port=3306,
       master_user='slave',
       master_password='slave123',
       master_log_file='mysql-bin.000001'
       master_log_pos=154;
  1. 开启主从
# Salve1、Slave2
mysql> start slave;
mysql> show slave status \G;

             Slave_IO_Running: Yes
            Slave_SQL_Running: Yes
              Replicate_Do_DB: backup

                  Master_Host: 192.168.0.3
                  Master_User: slave
                  Master_Port: 3306

              Master_Log_File: mysql-bin.000001
          Read_Master_Log_Pos: 154

                Relay_Log_Pos: 367
        Relay_Master_Log_File: mysql-bin.000001

            Slave_SQL_Running: Yes
              Replicate_Do_DB: backup
  1. 开启 MyCat
$ mycat start
总结

经过上面一系列到操作，最终恢复了数据和主从架构。虽然此方法可以恢复数据，但是前提是至少有一台 MySQL 节
点数据是未丢失到，只有满足这个前提，才能使用这种简单粗暴到方法。
